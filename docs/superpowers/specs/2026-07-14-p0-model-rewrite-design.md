# P0 核心模型重写设计

**日期：** 2026-07-14  
**状态：** 已批准  
**范围：** 天气、汇率、商品、CPI、每日 Tick、状态迁移与归档适配

## 1. 目标

将现有四个可单独运行、但未接入主引擎的 P0 原型，重写为可复现、可持久化、可校准的国家模拟基础框架。

完成后必须满足：

- 相同输入状态执行一次 Tick，得到完全相同的输出状态。
- 四个 P0 模型由主引擎真实调用，不再由旧版随机扰动代替。
- 所有跨日状态都显式写入 JSON，不依赖进程内隐藏变量。
- 旧版 `state.json` 可自动迁移，已有历史归档保持不变。
- 现有天气组件、Markdown 简报和 SQLite 摘要所需的兼容字段继续存在。
- 后续 GDP、人口、疾病和犯罪模型可以复用同一模型接口与随机流规则。

## 2. 非目标

- 不重写已有历史 JSON 归档或 SQLite 历史记录。
- 不在本次接入实时 IBTrACS 气旋数据。
- 不重新设计政府网站、气象网站、航空网站或仪表盘界面。
- 不修改死亡率、政治事件和其他非 P0 模型的业务假设；仅将它们迁移到确定性随机框架。
- 不引入第三方运行时依赖。

## 3. 架构原则

### 3.1 纯函数 Tick

主入口采用以下语义：

```python
next_state = tick(previous_state)
```

`tick` 不修改调用方传入的对象。它先复制并校验状态，将日期推进一天，然后依次执行模型，最后一次性返回完整的新状态。归档操作不属于 `tick`，仅在完整 Tick 成功后执行。

### 3.2 显式模型状态

每个模型统一接收：

- 当前模拟日期；
- 该模型上一日的 JSON 兼容状态；
- 只读上游输出；
- 为该模型和日期创建的独立随机流。

每个模型统一返回：

- 面向网站和其他模型的公开输出；
- 下一日需要持久化的模型状态；
- 当日产生的结构化事件。

模型不得依赖模块级可变变量、全局随机数或只存在于 Python 对象中的历史。

### 3.3 独立确定性随机流

随机流种子由 SHA-256 稳定派生：

```text
base_seed + schema_version + date + model_name + stream_name
```

使用 `random.Random(derived_seed)` 创建局部随机生成器。Python 内置 `hash()` 不参与派生，以避免不同进程之间的哈希随机化。模型调用顺序改变时，其他模型的结果不得改变。

### 3.4 依赖顺序

每日依赖图固定为：

```text
天气 ───────────────┐
汇率 ───────────────┼─> CPI ─> 国家状态合并 ─> 非 P0 事件 ─> 归档
商品 ───────────────┘
```

天气、汇率和商品彼此独立。CPI 只读取三者当日公开输出，不反向修改上游模型。

## 5. 状态 schema v2

顶层状态保留现有字段，并增加版本和显式模型状态：

```json
{
  "schema_version": 2,
  "date": "2026-08-11",
  "base_seed": 42,
  "weather": {},
  "economy": {},
  "population": 1200000,
  "government": {},
  "deaths_today": {},
  "events_today": [],
  "model_state": {
    "weather": {},
    "exchange": {},
    "commodities": {},
    "inflation": {}
  },
  "_meta": {}
}
```

### 5.1 天气兼容输出

`weather` 同时提供：

- 卡托拉兼容字段：`condition`、`temp_high`、`temp_low`、`humidity`、`rainfall_mm`、`wind_kmh`、`cyclone_risk`、`notes`；
- 五城明细：`katora`、`makadi_port`、`timo`、`pela`、`ruwa`；
- 国家指标：日出日落、ENSO、SOI、土壤湿度、火险、SST 和珊瑚白化风险。

### 5.2 经济兼容输出

`economy` 保留：

- `inflation_pct`；
- `exchange_rate_mvl_per_usd`；
- `fuel_95_price_mvl`；
- `fuel_diesel_price_mvl`；
- 现有 GDP、失业率和利率字段。

同时增加：

- `exchange_rates`：全部交叉汇率、锚点月份和陈旧天数；
- `commodities`：糖、黄金、Brent、来源月份和陈旧天数；
- `cpi`：指数水平、同比、环比、发布日期、分项通胀、分项贡献和至少 24 个月历史。

## 6. 状态迁移

`state.py` 根据 `schema_version` 判断状态格式。没有该字段的现有状态视为 v1。

迁移规则：

- 保留日期、人口、政府、死亡、事件和所有未知顶层字段。
- 将 `_meta.random_seed` 迁移为顶层 `base_seed`，缺失时使用 42。
- 用现有卡托拉天气初始化五城上一状态；缺失城市使用相同天气类别，不伪造历史降雨。
- 用现有 MVL/USD、通胀率和油价初始化经济模型状态。
- CPI 历史以当前同比通胀为锚点生成 13 个连续月度指数，使首次发布即可计算同比；历史条目标记为迁移基线。
- 商品模型状态记录数据文件中不晚于当前日期的最近观测月份。
- 迁移本身不推进日期，也不写文件。

迁移后立即执行完整 schema 校验。无法解释的日期、非数值经济指标或缺失核心对象会抛出带字段路径的 `StateValidationError`。

## 7. 天气模型

### 7.1 季节与温度

温度使用以 1 月 30 日为峰值、7 月 30 日为谷值的余弦周期：

```text
Tmean = 25.5 + 2.5 × cos(2π × (day_of_year - 30) / 365.2425)
```

城市偏移、海拔修正、云量降温和日较差在该基线之上计算。闰年使用实际年长，避免年末相位跳变。

### 7.2 天气状态

天气类别仍为晴、多云、阴、阵雨、暴雨。干湿季使用不同 Markov 转移矩阵，SOI 修正直接把概率质量在干类和湿类之间移动，再进行非负校验和归一化。

SOI 查找只允许使用不晚于模拟月份的最近真实观测；早于数据起点时返回中性状态，不得读取未来值。

### 7.3 降雨与风险

降雨使用标准库 `Random.gammavariate(shape, scale)`，参数以长期年降雨约 2,800 mm 为校准目标。滚动 7 日和 14 日降雨写入模型状态。

珊瑚白化判断先检查 `critical`，再检查 `warning`。气旋风险明确标记为气候统计风险，而非实时气旋观测。

## 8. 汇率模型

### 8.1 单位标准化

所有外汇数据在加载时转换为 `USD per currency unit`：

- AUD、NZD、EUR CSV 可直接使用；
- CNY CSV 的 `CNY per USD` 必须取倒数；
- USD 固定为 1。

交叉汇率统一计算为：

```text
MVL per currency = MVL per USD × USD per currency
```

### 8.2 货币篮子与每日过程

贸易权重保持 AUD 40%、NZD 25%、USD 20%、CNY 10%、EUR 5%。使用加权几何指数计算篮子相对基期的变化，避免不同报价尺度直接相加。

篮子产生当日目标 MVL/USD；显式状态的离散均值回归过程使昨日汇率向目标移动，并加入独立的小幅日冲击。边界仍为 1.80–2.80 MVL/USD。所有过程参数集中定义，并由长期测试验证稳定性。

真实月度数据使用严格的日历月份回退，不使用 30 天近似。

## 9. 商品模型

商品数据加载后按月份排序。查询使用二分查找取得“不晚于模拟月份的最后一个完整观测”。

输出包括：

- 糖价 USD/lb；
- 黄金 USD/oz；
- Brent USD/barrel；
- `source_month`；
- `staleness_days`；
- `is_stale`。

未来日期继续使用最后一个真实观测并明确标记陈旧。数据文件不存在、列缺失、月份重复冲突或找不到任何历史观测时抛出 `DataSourceError`，不提供硬编码市场价格。

## 10. CPI 模型

### 10.1 指标语义

CPI 模型明确区分：

- `index`：消费者价格指数水平；
- `mom_pct`：相对上月指数的环比；
- `yoy_pct`：相对 12 个月前指数的同比；
- 兼容字段 `economy.inflation_pct`：等于已发布的 `yoy_pct`。

### 10.2 频率与发布

模型每天积累成本压力，但官方 CPI 只在每月 15 日发布上一个完整日历月的数据。15 日之前显示上一次已发布值；发布后保持不变，直到下次发布。

同一日期重复执行相同输入不得产生第二次发布或不同结果。

### 10.3 分项联动

篮子权重保持：食品 35%、燃料 18%、住房 15%、交通 12%、其他 20%。

- 食品：使用滚动降雨相对逐日正常值的偏差、温度压力、糖价和汇率。
- 燃料：使用 Brent 与 MVL/USD，并通过税费和分销成本计算泵价。
- 住房：使用国家设定中的城市化率和建材进口汇率压力。
- 交通：使用燃料分项的滞后传导。
- 其他：使用低波动基础趋势和广义进口成本。

每月环比由分项贡献加权后更新指数。模型参数限制极端单月变化，限制必须作用于分项压力而不是事后伪造发布值。

## 11. 非 P0 事件迁移

现有死亡、政治事件和显著事件的概率与文案保持不变。它们迁移到 `events_model.py` 并使用按类别拆分的确定性随机流。天气警报读取新天气输出，不再使用七月硬编码天气。

## 12. 归档与命令行

### 12.1 原子归档

JSON 快照先写入同目录临时文件，完成刷新后使用 `os.replace` 原子替换。运行状态文件采用相同策略。

SQLite 的日摘要和事件写入位于一个事务中；任一写入失败时回滚。schema v2 的卡托拉兼容字段继续映射到现有 `daily_summary` 列，无需迁移历史表。

### 12.2 命令行

主入口支持：

```text
python engine/engine.py --days 1
python engine/engine.py --days 30
python engine/engine.py --days 365 --dry-run
```

`--days` 默认 1。`--dry-run` 执行完整模拟和校验但不写状态、归档或数据库，用于校准和人工检查。终端输出使用普通 Unicode 文本，不依赖无法在 GBK 编码的装饰符号。

## 13. 测试策略

### 13.1 单元测试

- 随机流：跨进程种子稳定；模型流互不干扰。
- 状态：v1→v2 迁移保留兼容字段；无效字段给出准确路径。
- 天气：1 月接近年度峰值、7 月接近谷值；SOI 不读取未来；Gamma 样本均值在容差内；critical 风险可达。
- 汇率：四个交叉盘单位正确；CNY 正确取倒数；日历回退不跳月；均值回归靠近目标。
- 商品：2026 日期返回 2024-12 真实数据和陈旧标记；缺失数据明确失败。
- CPI：15 日发布上月数据；发布日环比不恒为零；两次发布之间官方值保持不变；月正常降雨不被判为每日干旱。
- 事件：固定输入产生固定死亡和事件，不受 P0 调用顺序影响。

### 13.2 集成测试

- 主 Tick 实际调用四个 P0 模型，并生成 schema v2。
- 同一 v2 输入重复 Tick 结果完全相等。
- 连续运行与中途序列化、重新加载后继续运行的结果完全相等。
- 旧状态可直接运行第一个新 Tick。
- `--dry-run` 不改变任何文件。
- 归档在临时目录中写出兼容 JSON 和 SQLite 行；模拟写入失败时不留下半个状态。

### 13.3 365 天校准测试

固定种子运行完整年度，验证：

- 卡托拉年降雨量位于 2,240–3,360 mm；
- 湿季总降雨高于干季；
- 年度最高月平均温度出现在 12–2 月，最低出现在 6–8 月；
- 五城温度和降雨保持既定相对梯度；
- MVL/USD 始终处于 1.80–2.80；
- 官方 CPI 仅在发布日改变，且全年恰有 12 次月度发布；
- 所有数值为有限数，状态可被标准 JSON 编码。

## 14. 验收标准

实现完成的判定条件：

1. 全部标准库 `unittest` 测试通过且测试数大于零。
2. 五个核心 Python 模块通过 `py_compile`。
3. 365 天干运行成功，校准断言全部通过。
4. v1 状态迁移后的一日 Tick 成功，兼容字段可被天气组件和归档读取。
5. 全库不存在 P0 旧版 `roll_weather` 或经济随机扰动调用路径。
6. 默认 Windows PowerShell 环境运行命令行入口不发生编码异常。
7. 测试和干运行不修改版本库中的状态、归档或数据库。
